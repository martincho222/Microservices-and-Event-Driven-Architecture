# Patrón de Arquitectura Micro-frontends / Micro-frontends Architecture Pattern

---

## 1. Problemas del frontend monolítico / Problems of a Monolithic Frontend

**Español:**
Cuando una organización migra su backend a microservicios pero mantiene un único frontend monolítico, resuelve solo la mitad del problema. El frontend se convierte en el nuevo cuello de botella: todos los equipos compiten por el mismo repositorio, los despliegues requieren coordinación global y la velocidad de entrega se degrada exactamente igual que en el monolito original. El backend puede escalar y desplegarse de forma independiente, pero nada llega al usuario hasta que el frontend también se despliega — y ese deploy está bloqueado por la coordinación entre todos los equipos.

**English:**
When an organization migrates its backend to microservices but keeps a single monolithic frontend, it solves only half the problem. The frontend becomes the new bottleneck: all teams compete for the same repository, deployments require global coordination, and delivery speed degrades exactly as in the original monolith. The backend can scale and deploy independently, but nothing reaches the user until the frontend is also deployed — and that deployment is blocked by coordination across all teams.

```
El problema: microservicios en backend, monolito en frontend / The problem:

  ┌─────────────────────────────────────────────────────────────────┐
  │                FRONTEND MONOLÍTICO (React SPA)                  │
  │                                                                 │
  │  squad-catalog/   squad-orders/   squad-payments/  squad-users/ │
  │  ProductList.jsx  OrderHistory.jsx  Checkout.jsx   Profile.jsx  │
  │  ProductDetail    OrderStatus       PaymentForm     Settings    │
  │  CategoryNav      Cart.jsx          InvoiceView     AuthForms   │
  │                                                                 │
  │  → Todos en el mismo repo, mismo bundle, mismo deploy           │
  └─────────────────────────────────────────────────────────────────┘
                              │
            ┌─────────────────┼──────────────────┐
            ▼                 ▼                  ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │ Catalog Svc  │  │  Order Svc   │  │ Payment Svc  │
    │ (microserv.) │  │ (microserv.) │  │ (microserv.) │
    └──────────────┘  └──────────────┘  └──────────────┘

  → El backend tiene independencia total
  → El frontend anula esa independencia: un bug en Checkout bloquea
    el deploy de mejoras en ProductList de otro equipo
```

**Español:**
Los problemas concretos que genera un frontend monolítico en un entorno de microservicios son profundos y afectan tanto a la velocidad de desarrollo como a la calidad del producto final:

**English:**
The concrete problems generated by a monolithic frontend in a microservices environment are deep and affect both development speed and final product quality:

```
Problemas concretos / Concrete problems:

  ✗ Despliegues coordinados obligatorios
     → El equipo de Catalog no puede hacer release sin esperar que
       el equipo de Payments termine sus propias features
     → Un solo test roto en cualquier parte del frontend bloquea
       el deploy de todos los demás equipos
     → El tiempo entre "feature lista" y "feature en producción"
       crece con cada nuevo equipo que se suma al proyecto

  ✗ Stack tecnológico bloqueado para siempre
     → Si el proyecto empezó con Angular, todos los equipos usarán
       Angular indefinidamente, aunque React o Vue sean mejores para
       ciertos casos de uso
     → Imposible experimentar con nuevas tecnologías por equipo

  ✗ Build times interminables
     → Un bundle que incluye todo el frontend de ShopFast (catalog,
       orders, payments, users, search) tarda minutos en compilar
     → Los desarrolladores esperan para ver el resultado de un cambio
       de una sola línea

  ✗ Conflictos de CSS y estado global
     → Los estilos de un componente del equipo de Orders afectan
       accidentalmente la UI del equipo de Catalog
     → El estado global (Redux store, Context) se convierte en un
       campo minado donde cualquier equipo puede pisar el estado de otro

  ✗ Onboarding costoso
     → Un nuevo desarrollador debe entender la totalidad del frontend
       antes de poder contribuir sin riesgo a su área específica
     → El contexto necesario para trabajar en Checkout incluye entender
       cómo funciona el carrito, el catálogo y el perfil de usuario
```

> **ES:** El frontend monolítico invalida las promesas de los microservicios: si los equipos no pueden desplegar su parte del frontend de forma independiente, la autonomía del backend no se traduce en velocidad de entrega real para el negocio.
>
> **EN:** The monolithic frontend invalidates the promises of microservices: if teams cannot deploy their frontend piece independently, backend autonomy does not translate into real delivery speed for the business.

## 2. El patrón de arquitectura Micro-frontends / The Micro-frontend Architecture Pattern

**Español:**
El patrón de **micro-frontends** descompone la aplicación frontend en piezas más pequeñas, cada una alineada con un dominio de negocio o equipo específico. Cada micro-frontend es desarrollado, probado y desplegado de forma completamente independiente por el equipo que también posee el microservicio backend correspondiente. Una **App Shell** (o container application) es responsable de orquestar y componer los micro-frontends en la página que ve el usuario final.

**English:**
The **micro-frontends** pattern decomposes the frontend application into smaller pieces, each aligned with a specific business domain or team. Each micro-frontend is developed, tested, and deployed completely independently by the team that also owns the corresponding backend microservice. An **App Shell** (or container application) is responsible for orchestrating and composing the micro-frontends in the page the end user sees.

```
Arquitectura micro-frontend / Micro-frontend architecture (ShopFast):

         Usuario / User
              │
              ▼
  ┌─────────────────────────────────────────────────────────────┐
  │                    APP SHELL / CONTAINER                    │
  │          (routing, autenticación, layout global)            │
  └──────┬──────────┬──────────┬──────────┬────────────────────┘
         │          │          │          │
         ▼          ▼          ▼          ▼
  ┌──────────┐ ┌─────────┐ ┌────────┐ ┌──────────┐
  │ Catalog  │ │  Cart & │ │ User   │ │ Search   │
  │  MFE     │ │ Orders  │ │Profile │ │  MFE     │
  │          │ │  MFE    │ │  MFE   │ │          │
  │ React    │ │ Vue.js  │ │ React  │ │ Svelte   │
  │ v18      │ │ v3      │ │ v18    │ │ v4       │
  └────┬─────┘ └────┬────┘ └───┬────┘ └────┬─────┘
       │            │          │            │
       ▼            ▼          ▼            ▼
  Catalog Svc  Order Svc   User Svc    Search Svc
  (backend)    (backend)   (backend)   (backend)

  → Cada MFE tiene su propio repositorio, CI/CD y ciclo de deploy
  → Cada squad es dueño del MFE + microservicio de su dominio
```

### Estrategias de integración / Integration Strategies

**Español:**
Existen varias formas de componer los micro-frontends en la aplicación que ve el usuario. La elección depende del nivel de aislamiento necesario, la complejidad del routing y los requisitos de rendimiento:

**English:**
There are several ways to compose micro-frontends into the application the user sees. The choice depends on the level of isolation needed, routing complexity, and performance requirements:

```
Estrategia 1: Integración en runtime vía Module Federation (Webpack 5)
─────────────────────────────────────────────────────────────────────
  App Shell carga dinámicamente los MFEs desde sus propias URLs:

  // app-shell/src/bootstrap.js
  const CatalogMFE = await import('http://catalog.shopfast.com/remoteEntry.js')
  const OrdersMFE  = await import('http://orders.shopfast.com/remoteEntry.js')

  ✅ Pros: deploy independiente real, cada MFE sirve su propio bundle
  ❌ Contras: complejidad de configuración, múltiples round-trips de red

Estrategia 2: Integración en tiempo de build
─────────────────────────────────────────────
  Los MFEs se publican como paquetes npm y el App Shell los instala:

  // package.json del App Shell
  "dependencies": {
    "@shopfast/catalog-mfe": "^2.1.0",
    "@shopfast/orders-mfe":  "^1.8.0"
  }

  ✅ Pros: bundle optimizado, un solo request al cargar la app
  ❌ Contras: requiere redesploy del App Shell al actualizar cualquier MFE
              → los equipos no son 100% independientes en el deploy

Estrategia 3: Iframes
──────────────────────
  Cada MFE carga en un iframe independiente en la página:

  <iframe src="https://catalog.shopfast.com" />
  <iframe src="https://orders.shopfast.com" />

  ✅ Pros: aislamiento total (CSS, JS, estado) — imposible que un MFE
           afecte a otro
  ❌ Contras: UX degradada, rendimiento bajo, comunicación entre MFEs
              compleja, problemas de accesibilidad

Estrategia 4: Web Components
──────────────────────────────
  Cada MFE se expone como un Custom Element nativo del browser:

  <catalog-product-list category="electronics" />
  <orders-cart summary />

  ✅ Pros: estándar web nativo, funciona con cualquier framework
  ❌ Contras: compatibilidad limitada con frameworks SPA, SSR complejo
```

### Comunicación entre micro-frontends / Communication Between Micro-frontends

**Español:**
Los micro-frontends deben estar **lo más desacoplados posible** entre sí. La comunicación directa (llamadas entre componentes de distintos MFEs) crea el mismo acoplamiento que las librerías compartidas. El mecanismo preferido es el de **eventos del navegador** o **URL / query params**:

**English:**
Micro-frontends should be **as decoupled as possible** from each other. Direct communication (calls between components of different MFEs) creates the same coupling as shared libraries. The preferred mechanism is **browser events** or **URL / query params**:

```
Comunicación recomendada / Recommended communication:

  ❌ Acoplamiento directo / Direct coupling:
     CatalogMFE.addToCart(productId)  // MFE llamando directamente a otro MFE

  ✅ Custom Events (browser-native):
     // Catalog MFE emite:
     window.dispatchEvent(new CustomEvent('cart:add', {
       detail: { productId: '123', quantity: 1 }
     }))

     // Cart MFE escucha:
     window.addEventListener('cart:add', (event) => {
       cartService.addItem(event.detail)
     })

  ✅ URL compartida:
     // Catalog MFE navega a:
     /checkout?productId=123&quantity=1
     // Checkout MFE lee los query params al montar

  → Ningún MFE importa código del otro → desacoplamiento real
```

## 3. Ejemplo real: Micro-frontends + Microservicios / Real-life Example: Micro-frontends + Microservices

**Español:**
El ejemplo más claro de micro-frontends en producción a gran escala es **Amazon**. La página de detalle de un producto en Amazon es en realidad la composición de docenas de micro-frontends independientes: el carrusel de imágenes del producto, el bloque de precio y disponibilidad, las reseñas, las preguntas y respuestas, los productos relacionados, la sección de "Comprado junto con", el widget de shipping — cada uno desarrollado y desplegado por un equipo distinto. El usuario ve una página coherente; debajo hay una orquestación de componentes completamente independientes.

**English:**
The clearest example of micro-frontends in production at large scale is **Amazon**. The product detail page on Amazon is actually the composition of dozens of independent micro-frontends: the product image carousel, the price and availability block, reviews, Q&A, related products, the "Frequently bought together" section, the shipping widget — each developed and deployed by a different team. The user sees a coherent page; underneath is an orchestration of completely independent components.

```
ShopFast como ejemplo aplicado / ShopFast as an applied example:

  URL: shopfast.com/products/laptop-x1

  ┌──────────────────────────────────────────────────────────────────┐
  │  HEADER + NAV  ← App Shell (routing, auth, global layout)        │
  ├──────────────────────┬───────────────────────────────────────────┤
  │                      │  ┌────────────────────────────────────┐   │
  │  CATALOG MFE         │  │ PRICING MFE                        │   │
  │  → Imágenes          │  │ → Precio actual: $999              │   │
  │  → Descripción       │  │ → Descuentos activos               │   │
  │  → Especificaciones  │  │ → Stock: "Solo quedan 3"           │   │
  │                      │  └────────────────────────────────────┘   │
  │  Squad: Catalog      │  Squad: Pricing                           │
  │  Deploy: 4x/semana   │  Deploy: 10x/día                         │
  ├──────────────────────┴───────────────────────────────────────────┤
  │  REVIEWS MFE                         RECOMMENDATIONS MFE         │
  │  → Rating: 4.7★ (2,341 reseñas)     → "También te puede gustar"  │
  │  → Filtros por estrellas             → ML-powered suggestions     │
  │  Squad: Reviews                      Squad: ML/Personalization    │
  │  Deploy: 2x/semana                   Deploy: múltiples/día        │
  ├──────────────────────────────────────────────────────────────────┤
  │  CART MFE: [Añadir al carrito] [Comprar ahora]                   │
  │  Squad: Orders | Deploy: en cualquier momento, independientemente │
  └──────────────────────────────────────────────────────────────────┘

  Squad   │ MFE          │ Backend Svc    │ Deploy independiente
  ────────┼──────────────┼────────────────┼─────────────────────
  Catalog │ Catalog MFE  │ Catalog Svc    │ ✅ Sin coordinar
  Orders  │ Cart MFE     │ Order Svc      │ ✅ Sin coordinar
  Pricing │ Pricing MFE  │ Pricing Svc    │ ✅ Sin coordinar
  Reviews │ Reviews MFE  │ Reviews Svc    │ ✅ Sin coordinar
  ML      │ Recs MFE     │ Recommend. Svc │ ✅ Sin coordinar
```

**Español:**
La clave de este ejemplo es la alineación entre equipos, micro-frontends y microservicios. Cada squad es **dueño vertical** de su dominio: desde la base de datos hasta el pixel en la pantalla del usuario. Esto es lo que Pogrebinsky llama **full ownership** — el equipo que construye el servicio es el mismo que lo opera en producción y el mismo que construye la UI que lo consume.

**English:**
The key to this example is the alignment between teams, micro-frontends, and microservices. Each squad has **vertical ownership** of its domain: from the database to the pixel on the user's screen. This is what Pogrebinsky calls **full ownership** — the team that builds the service is the same one that operates it in production and the same one that builds the UI that consumes it.

## 4. Beneficios y buenas prácticas / Benefits and Best Practices

### Beneficios / Benefits

```
✅ 1. Despliegue verdaderamente independiente
      → El equipo de Reviews puede hacer deploy 10 veces al día sin
        bloquear al equipo de Catalog ni a ningún otro
      → Un bug en el MFE de Pricing no impide que el resto de la página
        funcione — el App Shell puede mostrar un fallback mientras se corrige

✅ 2. Libertad tecnológica por dominio
      → El equipo de Recommendations puede usar Svelte para su MFE de
        sugerencias interactivas mientras el resto del equipo usa React
      → Cada equipo puede actualizar su versión de framework sin
        coordinar con nadie
      → Posible migrar de Angular a React de forma incremental, MFE por MFE

✅ 3. Equipos autónomos de punta a punta (end-to-end)
      → Cada squad posee el stack completo: DB → API → MFE → UX
      → Alineado con la Ley de Conway: la arquitectura refleja la
        estructura del equipo, no al revés
      → Sin dependencias entre equipos para entregar una feature completa

✅ 4. Reemplaza el codebase monolítico complejo por micro-frontends pequeños y manejables
      (Replace the complex monolithic codebase with small and manageable micro-frontends)
      → El frontend monolítico de ShopFast tenía 200K líneas en un solo repo
        → imposible para un nuevo desarrollador entender el todo antes de contribuir
      → Con micro-frontends, el repositorio de Catalog MFE tiene ~5K líneas
        → un desarrollador nuevo es productivo en días, no en meses
      → Cada MFE tiene su propio package.json, sus propias dependencias
        y su propia configuración — sin interferencia con los demás

✅ 5. Full stack ownership de cada micro-frontend
      → El mismo squad que diseña la API del microservicio construye
        y mantiene la UI que la consume
      → No hay handoff entre "equipo de frontend" y "equipo de backend"
        — una sola conversación, un solo backlog, un solo responsable
      → Cuando hay un bug en producción que cruza la UI y el API,
        el mismo equipo lo entiende y lo corrige sin coordinación externa
      → Alineado con el principio de "you build it, you run it" de Pogrebinsky

✅ 6. Más fácil y rápido de testear en aislamiento
      (Easier / faster to test in isolation)
      → Cada MFE se puede testear completamente de forma independiente:
        unit tests, component tests e integration tests sin levantar
        el resto de la aplicación
      → Los mocks del backend son específicos del dominio del MFE —
        no hay que simular toda la API de la aplicación
      → Los test suites son más pequeños → feedback loop más rápido
        para el desarrollador (segundos vs minutos)
      → Una falla en los tests de Cart MFE no cancela el pipeline
        de Catalog MFE — cada equipo ve solo sus propios resultados

✅ 7. Pipeline de CI/CD separado por micro-frontend
      (Separate CI/CD pipeline per micro-frontend)
      → Cada MFE tiene su propio pipeline: build → test → deploy
      → Un deploy de Reviews MFE no requiere re-compilar ni re-testear
        el bundle de Orders MFE — tiempo total de pipeline: minutos
      → Si el pipeline de Catalog MFE falla, el resto de los equipos
        siguen desplegando normalmente sin ningún impacto
      → Posible tener cadencias de deploy radicalmente distintas
        por equipo según las necesidades del negocio:
        Pricing MFE: 10+ deploys/día  (precios en tiempo real)
        Reviews MFE: 1-2 deploys/semana (feature development más lento)

✅ 8. Codebases más pequeñas y manejables
      → Un desarrollador nuevo en el equipo de Orders solo necesita
        entender el repositorio de Orders MFE — no los 200K líneas
        del frontend monolítico
      → Build times en segundos en lugar de minutos
      → Tests más rápidos y enfocados en el dominio del equipo

✅ 9. Resiliencia y degradación elegante
      → Si el MFE de Recommendations falla, el usuario aún puede ver
        el producto, ver preguntas y comprar
      → El App Shell puede implementar circuit breakers para cada MFE
      → Un error en producción afecta un área acotada, no toda la app
```

### Desventajas / Downsides

```
❌ 1. Complejidad de orquestación
      → El App Shell debe gestionar routing, autenticación, y carga de MFEs
      → La integración de múltiples MFEs (especialmente Module Federation)
        tiene una curva de configuración pronunciada
      → El equipo de Platform necesita mantener la infraestructura de
        composición — un costo nuevo que no existía con el SPA monolítico

❌ 2. Rendimiento: múltiples bundles
      → Con Module Federation, el usuario descarga múltiples JavaScript
        bundles desde distintos orígenes (catalog.shopfast.com, orders.shopfast.com)
      → Sin una estrategia de caché agresiva, el tiempo de carga inicial
        puede ser peor que con un SPA monolítico bien optimizado
      → Requiere HTTP/2, code splitting y lazy loading deliberados

❌ 3. Consistencia de UX y diseño
      → Si cada equipo construye sus componentes de forma independiente,
        la UI puede verse inconsistente entre secciones
      → Solución: Design System compartido (Tier 3 del modelo de autonomía)
        — un repositorio de componentes visuales que todos los MFEs consumen
      → El Design System es un caso donde el código compartido SÍ está
        justificado (es infraestructura de UI, no lógica de negocio)

❌ 4. Testing de integración complejo
      → Probar que el MFE de Cart y el MFE de Catalog interactúan
        correctamente requiere un entorno donde ambos estén desplegados
      → Los tests end-to-end deben coordinar múltiples repositorios y
        versiones independientes de cada MFE

❌ 5. Duplicación de dependencias
      → Si Catalog MFE y Orders MFE usan React, el usuario puede terminar
        descargando dos versiones de React (si no se configura correctamente)
      → Requiere configuración de shared dependencies en Module Federation
        para evitar duplicación
```

### Buenas prácticas / Best Practices

```
  1. App Shell liviano / Lightweight App Shell
     → El App Shell solo debe encargarse de: routing global, autenticación,
       layout (header/footer), y carga de MFEs
     → NO debe contener lógica de negocio — si lo hace, se convierte en
       un monolito disfrazado

  2. Design System compartido como Tier 3
     → Definir componentes visuales base (botones, inputs, tipografía,
       colores) en una librería de UI compartida
     → Esta librería cambia muy raramente — su versioning es estable
     → Es el único código de UI que justifica ser compartido entre MFEs

  3. Contratos de eventos explícitos
     → Documentar todos los Custom Events que un MFE emite y consume
     → Tratarlos como APIs públicas: versionar, mantener backward compatibility
     → Ejemplo: 'cart:add' v1 → 'cart:item-added' v2 con período de transición

  4. Boundaries alineados con dominios de negocio
     → No crear un MFE por componente visual (eso son nano-frontends)
     → Un MFE = un subdominio de negocio = un equipo
     → La granularidad correcta: "¿puede un squad completo ser dueño de esto?"

  5. Fallbacks y error boundaries
     → Cada MFE debe ser cargado dentro de un error boundary del App Shell
     → Si un MFE lanza un error o tarda más de N segundos en cargar,
       mostrar un fallback — la página sigue siendo funcional

  6. Shared nothing (excepto Design System)
     → No compartir estado entre MFEs a través de variables globales
     → No importar código de un MFE en otro MFE directamente
     → La comunicación solo a través de eventos o URL — igual que
       microservicios se comunican solo por APIs o eventos

  7. Los micro-frontends se cargan en runtime / Micro-frontends are loaded at runtime
     → Cada MFE es un bundle independiente servido desde su propia URL
       y cargado dinámicamente por el App Shell cuando se necesita
     → Esto significa que un equipo puede hacer deploy de su MFE y el
       usuario recibe la nueva versión en su próxima navegación —
       sin que el App Shell ni ningún otro MFE haya sido redesploy
     → Implementado típicamente con Module Federation (Webpack 5) o
       import maps — el App Shell solo conoce la URL del entry point,
       no la versión ni el contenido del bundle
     → Consecuencia: los MFEs deben ser backward compatible con el
       App Shell — no se puede hacer un breaking change silencioso

  8. Sin estado compartido en el browser / No shared state in the browser
     → Cada MFE gestiona su propio estado de forma completamente aislada
     → Está prohibido usar variables globales (window.mySharedState),
       localStorage como canal de comunicación o un store de Redux global
       que todos los MFEs lean y escriban
     → Compartir estado entre MFEs crea el mismo acoplamiento oculto
       que una librería compartida — si un MFE cambia la estructura del
       estado, todos los que lo leen se rompen
     → Si dos MFEs necesitan los mismos datos, cada uno los obtiene
       directamente de su propio backend — no del estado del otro MFE

  9. Intercomunicación a través de mecanismos estándar
     (Intercommunication through: custom events, callbacks, address bar)

     a) Custom Events:
        → Para notificaciones de una sola dirección (fire-and-forget)
        → Catalog MFE dispara 'cart:add', Cart MFE escucha — sin acoplamiento directo
        → El contrato es el nombre del evento y la estructura del payload
        → window.dispatchEvent(new CustomEvent('cart:add', { detail: { ... } }))

     b) Callbacks:
        → El App Shell pasa funciones como props a cada MFE al montarlo
        → Permite que el App Shell coordine comportamiento entre MFEs
          sin que los MFEs se conozcan entre sí
        → Ejemplo: App Shell pasa onLoginSuccess() a Auth MFE
          → cuando el usuario hace login, Auth MFE invoca el callback
          → App Shell actualiza el estado de autenticación global
          → notifica a los demás MFEs vía evento o re-render

     c) Address bar (URL / query params / hash):
        → El mecanismo más simple y más robusto — la URL es el estado compartido
          universal que todos los MFEs pueden leer sin acoplarse entre sí
        → Catalog MFE navega a /checkout?items=123,456
          → Checkout MFE lee los query params al montar
        → Ventaja adicional: el estado es bookmarkable, shareable y
          sobrevive a un refresh de página — ningún otro mecanismo lo garantiza
```

> **ES:** Los micro-frontends no son para todos. Son la solución correcta cuando el principal cuello de botella de entrega es la coordinación entre equipos en el frontend. Para un proyecto con un solo equipo o menos de 5 desarrolladores en frontend, la complejidad operacional supera los beneficios. A mayor escala de equipo, mayor el valor.
>
> **EN:** Micro-frontends are not for everyone. They are the right solution when the main delivery bottleneck is team coordination on the frontend. For a project with a single team or fewer than 5 frontend developers, the operational complexity outweighs the benefits. The larger the team scale, the greater the value.
